# Working Preferences

## Working Standards and Style

- **Language:** Python 3
- **Code style:** snake_case for variables and functions, PascalCase for classes
- **Formatting:** Black (or similar auto-formatter) preferred — consistent formatting over personal style
- **Type hints:** Use where practical; at minimum on public function signatures
- **Naming conventions:** Descriptive names over short abbreviations; CLI commands use kebab-case
- **Testing expectations:** Write tests for core logic; use pytest
- **Git workflow:** Descriptive commit messages; small, focused commits

## Communication Preferences

- **Response style:** Direct and concise — get to the point, skip filler
- **When stuck:** Make a best-guess decision, note the assumption, and proceed
- **Planning:** For small tasks, just do it. For large or risky changes, show a plan first
- **Code explanations:** Briefly note what changed and why; skip obvious details

## How I Like to Work with AI

- **Autonomy level:** Proactive on small tasks; ask or show a plan before large refactors or architectural changes
- **Scope discipline:** Minimal changes only — don't touch code unrelated to the task
- **Decision making:** Pick the best straightforward approach; flag trade-offs only when they matter
- **Wiki updates:** Offer to update the wiki after tasks that yield durable knowledge; wait for approval before writing

## Preferences to Avoid

- Don't add unrequested features or abstractions
- Don't rewrite working code without a clear reason
- Don't use overly verbose explanations
- Don't add error handling for scenarios that can't happen
- Don't start responses with "Great!", "Certainly!", or similar filler phrases
